# LangGraph 工作流编排与输出节点设计

## 1. 背景与目标

当前工作流画布已具备节点放置、连线、配置保存和运行调试入口，但后端运行仍是固定的 LangChain 单模型流程，无法按画布结构执行。本次需要同时完成两部分能力：

- 新增可从左侧工具栏添加的“输出”节点，支持配置必填输出变量。
- 使用 LangGraph 将已保存的画布节点和连线动态编译为可执行工作流，支持条件分支、并行汇合和受限循环。

目标是建立前端配置、持久化、图校验、LangGraph 编译、运行追踪和最终输出之间的真实闭环。

## 2. 范围

### 2.1 本次实现

- 输出节点的添加、放置、连线、配置、校验和保存。
- 用户输入、知识检索、LLM、输出节点的真实 LangGraph 执行。
- 条件分支节点、并行扇出与汇合、受限循环节点。
- 节点级运行追踪、失败信息和最终输出。
- 画布保存及运行前的图结构与变量引用校验。

### 2.2 不在本次范围

- 人工审批后的跨请求暂停与恢复。
- 长期持久化检查点和会话记忆。
- 任意 Python 表达式、脚本或不受限循环。
- 新增工具调用、代码执行等其他业务节点。

## 3. 总体架构

采用“节点注册表 + 图校验器 + LangGraph 动态编译器”结构：

1. `WorkflowNodeRegistry` 维护每类节点的配置校验、输出变量声明和运行处理器。
2. `WorkflowGraphValidator` 校验入口、输出、边、分支、循环和变量引用。
3. `WorkflowGraphCompiler` 将持久化节点和边转换为 `StateGraph`，并编译为可执行图。
4. `WorkflowGraphExecutor` 负责注入初始状态、执行图、收集节点追踪并生成最终结果。
5. `AgentRunService` 保留现有运行接口，改为按智能体加载已保存工作流并委托执行器运行。

前端继续使用现有 React Flow 画布和工作流保存接口，不引入第二套画布状态模型。

## 4. 统一运行状态

LangGraph 状态使用统一结构：

- `inputs`：用户输入节点产生的命名变量。
- `node_outputs`：以节点 ID 为命名空间保存各节点输出。
- `final_output`：输出节点生成的最终返回值。
- `trace_steps`：节点执行记录，使用列表合并 reducer 支持并行写入。
- `route_decisions`：条件节点的分支选择结果。
- `iteration_counts`：循环节点的执行次数。

并行节点更新 `node_outputs` 时使用字典合并 reducer，避免并发分支互相覆盖。

## 5. 输出节点

### 5.1 添加与画布表现

- 左侧“添加节点”菜单增加“输出”。
- 点击后进入待放置状态，再点击画布生成 `type: "expose"`、名称为“输出”的节点。
- 输出节点是终点节点，仅显示左侧输入连接点，不显示右侧输出连接点。
- 节点卡片展示已配置的输出变量摘要；未配置时展示引导文案。

### 5.2 配置面板

- 面板标题直接显示“输出”。
- 不显示“设置”“上次运行”标签，也不显示通用运行摘要。
- “输出变量”标记为必填，点击“＋”新增空白变量行。
- 每行包含变量名、变量值和删除按钮。
- 变量名与变量值均为必填；变量名在当前输出节点内必须唯一。
- 配置保存为 `config.outputVariables`，每项结构为 `{ id, name, value }`。

### 5.3 上游变量范围

输出变量选择器展示当前输出节点所有递归可达的上游节点变量：

1. 从输出节点沿入边反向遍历。
2. 使用 `visited` 集合避免循环图无限遍历。
3. 排除当前输出节点、注释节点和不产生业务数据的纯控制节点。
4. 用户输入节点暴露其 `inputFields`。
5. LLM 节点暴露 `text`、`reasoning_content`、`usage`。
6. 知识检索节点暴露 `result`。
7. 其他可执行节点从节点注册表读取输出声明。

候选项由当前 `nodes` 和 `edges` 派生，不另存副本。新增、删除或改接连线后立即重算；没有可达上游变量时，下拉不展示变量选项。

## 6. 复杂编排

### 6.1 普通边与并行

- 单一出口编译为普通边。
- 同一节点的多个普通出口在同一 LangGraph superstep 执行，形成并行扇出。
- 多个前驱汇入同一节点时，编译器按前驱集合建立等待边，全部前驱完成后执行汇合节点。
- 并行分支通过 reducer 合并输出和追踪。

### 6.2 条件分支

条件节点使用结构化规则，不执行任意代码。配置包含：变量引用、比较运算符、比较值和分支 ID。支持等于、不等于、包含、大于、小于、为空和非空。

条件节点的出边使用 `sourceHandle` 标识分支，编译为条件边。未命中显式规则时走默认分支；配置缺少默认分支时校验失败。

### 6.3 循环

循环节点包含继续条件、循环体出口、退出出口和 `maxIterations`。每次进入循环节点递增计数；达到上限后强制退出并记录警告。后端同时设置 LangGraph 全图递归上限，形成双重保护。

循环允许引用当前循环路径中已经产生的变量，但输出节点的上游变量遍历仍通过 `visited` 保证有限计算。

## 7. 节点执行语义

- 用户输入：把请求数据映射到 `inputs`。
- 知识检索：按配置的知识库执行检索，输出 `result`。
- LLM：解析配置中的变量引用，调用对应模型，输出 `text`、`reasoning_content` 和 `usage`。
- 条件：读取状态并返回分支选择，不产生业务输出。
- 并行汇合：由图边和 reducer 表达，不需要额外业务处理器。
- 循环：计算继续或退出路径并维护次数。
- 输出：解析 `outputVariables`，生成结构化结果；仅一个变量时兼容生成字符串 `finalOutput`，多个变量时生成 JSON 字符串并保留结构化结果。
- 注释：编译时忽略。

## 8. API 与运行链路

保留 `POST /api/agents/{agent_id}/runs`：

1. 根据智能体查找已保存工作流。
2. 校验并编译 LangGraph。
3. 执行图并收集节点追踪。
4. 将状态、耗时、错误和最终输出写入现有运行追踪仓储。

无工作流、工作流无输出节点或图校验失败时，返回明确错误，不再静默退回固定单模型流程。

## 9. 校验与错误处理

保存和运行前校验：

- 恰好一个用户输入入口。
- 至少一个输出节点。
- 所有可执行边的两端节点存在。
- 输出节点不存在出边。
- 输出变量非空、名称唯一、引用有效且来自递归可达的上游节点。
- 条件分支目标完整，并存在默认分支。
- 循环上限处于允许范围，建议默认 10、最大 100。
- 不支持的节点类型不能进入运行图。

编译错误返回图级错误；执行错误定位到具体节点并写入失败步骤。并行分支任一失败时本次运行失败，不输出不完整结果。

## 10. 测试策略

### 10.1 前端

- 从工具栏添加并放置输出、条件和循环节点。
- 输出节点只有输入连接点。
- 输出变量增删、必填、名称唯一和持久化。
- 递归上游变量计算，包括多层链路、分支汇合、断开连线和循环图。
- 无可达上游时不展示变量选项。

### 10.2 后端

- 节点注册表和图校验器单元测试。
- 线性流程、条件分支、并行扇出/汇合、有限循环的编译与执行测试。
- 多分支状态 reducer 合并测试。
- 输出变量解析和最终输出测试。
- 无输出、悬空边、非法变量、缺失默认分支和循环超限测试。

### 10.3 端到端

保存画布后通过智能体运行接口执行真实工作流，验证节点追踪顺序、并行分支结果、循环次数和最终输出。

## 11. 实施分段

1. 输出节点、递归变量选择、图校验器、LangGraph 依赖和编译器骨架。
2. 当前业务节点真实执行及运行接口切换。
3. 条件分支与并行汇合。
4. 受限循环、错误追踪和完整端到端验证。

各段保持同一状态契约和节点注册表，不做临时固定流程。
