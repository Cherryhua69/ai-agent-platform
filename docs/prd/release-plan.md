# 版本发布计划

## 1. 发布策略

发布计划围绕 MVP 主路径逐步收敛：

创建智能体 -> 编排能力 -> 绑定 MCP 工具和知识库 -> 调试运行 -> 评测 -> 发布 -> 观测。

每个里程碑都应有可演示、可验收的闭环结果。M0 验证产品原型和关键信息架构，M1 完成 MVP 闭环，M2 增强企业治理和可运营能力。

## 2. M0 原型验证

### 目标

验证核心对象、页面结构和主流程是否能被目标用户理解，降低后续工程实现返工风险。

### 建议范围

- 工作台信息结构原型。
- 智能体列表、详情、创建草稿流程。
- 工作流编排器原型，包括节点库、画布、节点配置面板和调试结果面板。
- MCP Server 和工具详情原型。
- 知识库创建、文档导入、检索测试原型。
- Trace 详情原型。
- 评测和发布检查流程原型。
- 基础角色权限说明和关键操作清单。

### 不包含

- 完整后端接口实现。
- 真实模型调用、真实向量检索和真实 MCP 调用。
- 高级审计、企业 IM 渠道、模板市场。

### 关键验收

- AI 应用开发者可以理解如何创建智能体、配置工作流并查看 Trace。
- 运营人员可以理解如何维护知识库、测试集并触发发布前评测。
- 管理员可以理解空间、角色、工具密钥和权限边界。
- 原型能完整演示“创建智能体到发布测试环境”的主路径。
- 团队确认 P0/P1/P2 优先级和领域模型无重大冲突。

### 主要交付物

- 可点击产品原型或低保真页面流。
- P0 功能清单确认版。
- 用户故事确认版。
- 发布门禁规则草案。

### 风险与退出条件

- 风险：工作流编排能力过大，MVP 工程成本不可控。
- 风险：评测指标和发布门禁规则不够明确，影响闭环验收。
- 退出条件：目标用户完成原型走查，主流程无阻断级需求歧义。

## 3. M1 MVP 闭环

### 目标

交付最小可用产品，使团队可以从创建智能体到发布测试环境，并能通过运行日志诊断问题。

### P0 范围

#### 工作台

- 平台健康指标总览。
- 待处理事项入口，包括运行异常、待发布草稿、评测失败项、MCP 工具异常。

#### 智能体管理

- 智能体列表、创建、编辑、归档。
- 基础模型配置、系统提示词、变量和上下文策略。
- 智能体详情，展示配置、版本、绑定资源和运行概况。
- 草稿版本与已发布版本区分。

#### 工作流编排

- 可视化节点画布。
- 支持开始、LLM、条件分支、工具调用、知识库检索、结束节点。
- 节点参数配置。
- 保存草稿和运行当前草稿。
- 节点级运行结果和错误信息。

#### MCP / 工具

- 注册 MCP Server。
- 连接检查和工具列表同步。
- 查看工具 schema、输入输出参数和状态。
- 工具节点可选择授权工具。
- 工具调用日志和异常状态。

#### 知识库

- 创建知识库。
- 至少一种文档导入方式。
- 配置基础分段和检索参数。
- 检索测试。
- 绑定知识库到智能体或工作流节点。

#### 调试与运行日志

- 单轮调试。
- Run 和 RunTrace 记录。
- Trace 详情展示提示词、检索、模型、工具、输出、token、耗时、成本和错误。
- 运行列表与基础筛选。

#### 评测

- 创建测试集和测试用例。
- 关键用例标记。
- 批量运行评测。
- 展示通过率、平均延迟、平均成本和工具调用成功率。
- 发布前门禁阻断关键测试失败版本。

#### 发布渠道

- 发布检查。
- 发布到测试环境。
- API 渠道基础配置，或 Web Chat/API 至少完成一个可用渠道。
- Release 记录。

#### 权限与空间

- 空间与项目边界。
- Owner、Admin、Developer、Operator、Viewer 基础角色。
- 列表、详情和资源选择器按授权范围过滤。

### 建议验收场景

1. 管理员创建空间和项目，配置一个 MCP Server 并同步工具。
2. 开发者创建智能体草稿，配置模型和系统提示词。
3. 开发者在工作流中添加 LLM、知识库检索、MCP 工具和条件分支节点。
4. 运营创建知识库、导入文档并完成检索测试。
5. 开发者运行草稿调试，并在 Trace 中看到节点输入、输出、耗时和错误。
6. 运营创建测试集，运行批量评测。
7. 关键用例通过后，运营将智能体发布到测试环境。
8. 发布后，团队可以在运行日志中查看调用记录和 Trace。
9. 当工具异常或关键评测失败时，发布检查给出明确阻断原因。

### 成功指标

- 从创建智能体到发布测试环境的路径可闭环。
- P0 功能无阻断级缺口。
- 每次运行可追踪、可诊断、可复现。
- MCP 工具和知识库作为一等资源被管理和绑定。
- 非技术运营人员可以完成知识库、测试集和发布检查相关配置。

### 不包含

- 完整插件市场。
- 多租户商业计费。
- 自动化智能体生成。
- 复杂 Agent-to-Agent 协作。
- 高级灰度发布和 A/B 实验。
- 完整企业 SSO。
- 企业微信、飞书、钉钉等企业 IM 正式接入。

### 风险与缓解

- 风险：真实 MCP Server 连接和 schema 同步复杂度高。
  - 缓解：M1 先支持有限 transport 和清晰的 schema 校验错误。
- 风险：知识库导入和向量检索质量影响用户感知。
  - 缓解：M1 至少保证一种稳定导入方式，并在检索测试中展示命中依据。
- 风险：评测门禁定义过宽或过窄。
  - 缓解：M1 优先采用关键用例通过/失败和工具状态作为硬门禁。
- 风险：权限模型复杂导致开发周期延长。
  - 缓解：M1 先实现空间、项目和基础角色边界，资源级权限放入 P1。

## 4. M2 企业治理增强

### 目标

在 MVP 闭环稳定后，补齐企业团队在权限、审计、发布风险、运营分析上的增强能力。

### 建议范围

#### 资源级权限

- 智能体级、工具级、知识库级授权。
- 工具选择器、知识库选择器和发布检查读取资源级权限。

#### 审计日志

- 记录发布、回滚、密钥变更、权限变更、工具配置变更。
- 支持按操作类型、资源、操作者和时间筛选。

#### 发布治理

- 回滚到历史版本。
- 发布检查增强，包括工具状态、知识库状态、关键评测结果和权限配置。
- 环境级发布策略预留。

#### 运营分析

- 工作台趋势指标。
- 按智能体、渠道、环境查看成功率、延迟、成本趋势。
- 高频错误聚合。

#### 评测增强

- 人工评分。
- 评测报告汇总。
- 指标阈值配置预留。

#### 渠道增强

- Web Chat 渠道配置完善。
- 企业 IM 渠道作为预留入口，不作为 M2 必须正式接入项。

### 关键验收

- 管理员可以追溯关键高风险操作。
- 资源级权限能影响工具、知识库和智能体访问。
- 运营可以通过趋势和评测报告判断版本质量变化。
- 已发布版本出现问题时可以回滚到历史版本。
- 发布检查能呈现可操作的阻断原因和跳转入口。

### 成功指标

- 高风险操作均产生审计记录。
- 发布失败或阻断原因可解释、可追踪。
- 权限变更后，未授权资源不会出现在列表、详情和选择器中。
- 运营能够定位主要失败来源和成本/延迟变化。

## 5. 后续版本候选方向

- 企业 IM 渠道正式接入，包括企业微信、飞书、钉钉。
- 高级模型路由和成本策略。
- 模板市场和智能体模板化创建。
- 高级灰度发布、A/B 实验和流量分配。
- 完整企业 SSO。
- 自动化智能体生成和复杂 Agent-to-Agent 协作。
