# 工作流画布设计方向

## 1. 背景与目标

当前平台已经具备智能体列表、工作流配置页、React Flow 画布、基础节点保存、节点删除、运行调试等能力。下一阶段的目标不是简单复刻 Dify，而是在参考 Dify 工作流编排体验的基础上，形成适合本项目的智能体工作流画布：用户从智能体卡片进入对应工作流，围绕“用户输入 -> 节点处理 -> 直接回复/输出”的数据流完成配置、连线、保存与测试。

设计目标如下：

- 让非工程用户能通过拖拽、点击、连线完成智能体工作流搭建。
- 让工程实现有稳定的数据模型，节点、边、配置、视口、运行结果都能持久化。
- 当前聚焦 LLM 节点，把交互、保存、调试链路打通；后续平滑扩展检索、工具、人审、条件、循环等节点。
- 画布体验参考 Dify 的“主画布 + 左侧工具栏 + 右侧节点配置 + 手动连线”结构，但保留本项目更偏企业智能体平台的冷静、专业、信息密度适中的视觉风格。

## 2. Dify 调研结论

Dify 的 Workflow 和 Chatflow 都建立在共享的可视化画布和节点系统之上，用户通过拖拽节点、连接节点、配置节点来编排模型调用、知识检索、工具调用、条件分支等流程。参考资料：Dify Workflow & Chatflow 文档，https://docs.dify.ai/en/use-dify/build/workflow-chatflow。

Dify 的关键设计启发：

- 工作流由节点和连线构成，每个节点负责一个明确步骤，例如模型调用、知识检索、代码执行、条件判断或回答输出。
- 节点可以串行或并行执行。串行节点能读取上游节点变量；并行节点互相不能直接读取同级变量，但汇合后的下游节点可以读取多个分支结果。参考资料：Dify Orchestration Logic，https://docs.dify.ai/en/use-dify/build/orchestrate-node。
- Start/User Input 节点是工作流入口，负责收集用户输入变量；Dify 还区分手动触发的 User Input 和自动触发的 Trigger。参考资料：Dify Key Concepts，https://docs.dify.ai/en/use-dify/getting-started/key-concepts。
- LLM 节点是核心处理节点，调用配置好的模型，支持提示词、上下文、多模态输入、结构化输出等能力。参考资料：Dify LLM Node，https://docs.dify.ai/en/use-dify/nodes/llm。
- Knowledge Retrieval 节点负责从知识库检索内容，典型路径是 User Input -> Knowledge Retrieval -> LLM -> Answer。参考资料：Dify Knowledge Retrieval，https://docs.dify.ai/en/use-dify/nodes/knowledge-retrieval。
- Answer/Output 节点用于把上游变量组合成最终返回内容，支持文本、变量引用、图片、文件等输出。参考资料：Dify Answer Node，https://docs.dify.ai/en/use-dify/nodes/answer；Dify Output Node，https://docs.dify.ai/en/use-dify/nodes/output。
- 条件、循环、迭代、工具、人类输入是高级编排能力，可作为后续节点类型扩展。参考资料：If-Else、Iteration、Tool Node、Human Input 文档。

对本项目的取舍：

- 第一阶段不做完整 Dify 节点库，避免节点多但不可运行。先把用户输入、LLM、直接回复、注释框四类体验做扎实。
- 当前画布必须支持手动连线，因为连线就是数据流转关系，不应由系统暗中自动推断。
- 节点配置面板默认关闭，点击节点后打开，避免进入页面时右侧配置干扰用户观察整体流程。
- 保存按钮必须真实持久化用户配置，包括节点、边、节点位置、节点配置、视口状态。

## 3. 画布总体布局

工作流配置页由四个区域构成：

- 顶部栏：左侧显示智能体名称和描述，右侧显示保存、运行调试。不要显示独立“工作流”导航标题、模型状态、预览按钮。
- 主画布：占满页面主卡片空间，是用户工作的核心区域。画布背景使用浅色网格点阵，保留缩放、拖拽、节点选择、连线。
- 左侧工具栏：悬浮在画布左侧，提供添加节点、添加注释框、指针模式、手模式、自动整理节点五个功能。
- 右侧节点配置面板：默认关闭。点击节点后打开，显示该节点的配置表单、运行摘要、关闭按钮。用户可以手动关闭，关闭后不影响画布选择状态和保存状态。

页面信息层级：

1. 用户首先看到智能体名称、描述和整体流程。
2. 用户在画布中选择节点、拖动节点、连接节点。
3. 用户需要配置时点击节点打开右侧详情。
4. 用户保存或运行调试时从顶部栏完成操作。

## 4. 左侧工具栏设计

工具栏定位为画布操作工具，不是页面导航。建议保持窄栏、图标优先、hover tooltip 辅助说明，风格上与现有冷静 SaaS 界面一致。

五个工具如下：

| 顺序 | 功能 | 图标方向 | 行为 |
| --- | --- | --- | --- |
| 1 | 添加节点 | 加号叠加节点/模块感图标 | 点击展开节点菜单，目前菜单只包含 LLM，后续扩展检索、工具、条件等 |
| 2 | 添加注释框 | 对话框/便签/注释图标 | 进入待放置状态，用户点击画布位置后创建注释框 |
| 3 | 指针模式 | 指针箭头 | 节点可选中、拖拽、连线，是默认模式 |
| 4 | 手模式 | 手掌 | 画布平移优先，节点不被误拖动 |
| 5 | 自动整理节点 | 闪光/排列/魔法棒图标 | 按当前数据流方向重新排列节点，尽量保持从左到右 |

交互规则：

- 工具栏按钮尺寸固定，避免 hover 或 active 时布局跳动。
- 激活态使用项目主色描边或淡色背景，不使用过度发光。
- “添加节点”点击后弹出节点菜单，菜单贴近按钮右侧，展示节点名称、类型说明、图标。
- 选择“添加 LLM 节点”后，不立即放到固定位置，而是进入待放置状态。用户点击画布后，节点出现在点击位置。
- 待放置状态需要 toast 或轻提示，例如“点击画布放置 LLM 节点”。
- 添加注释框与添加节点一致，用户决定落点。

## 5. 节点体系

### 5.1 当前需要落地的节点

当前画布实际应包含四类节点：

| 节点 | 类型 | 是否默认 | 是否可删除 | 是否参与数据流 | 说明 |
| --- | --- | --- | --- | --- | --- |
| 用户输入 | `trigger` | 是 | 否 | 是 | 每个智能体必须有的入口节点 |
| LLM | `llm` | 是，可新增 | 是 | 是 | 当前唯一可新增的工具节点 |
| 直接回复 | `expose` | 是 | 可按产品策略决定，建议后续可删但需至少一个输出 | 是 | 把上游结果返回给用户 |
| 注释框 | `comment` | 否 | 是 | 否 | 画布说明，不参与运行 |

默认智能体创建后，建议生成基础链路：

```mermaid
flowchart LR
  A["用户输入"] --> B["LLM"]
  B --> C["直接回复"]
```

### 5.2 用户输入节点

用户输入节点是工作流入口，每个智能体必须拥有且只拥有一个默认用户输入节点。它不能被删除，应该始终排在画布数据流最左侧。

配置项：

- 用户文字输入：用于运行调试和变量引用，保存为 `userTextInput`，兼容旧字段 `userInput`。
- 用户文件上传输入：用于表示用户上传的文件变量或文件名，保存为 `userFileUploadInput`。

后续扩展：

- 支持多个输入字段，例如文本、文件、数字、下拉、多选。
- 输入字段作为变量暴露给下游节点，例如 `{{user_input.text}}`、`{{user_input.files}}`。
- 用户输入节点可配置“是否必填”“默认值”“占位提示”“文件类型限制”。

### 5.3 LLM 节点

LLM 节点是当前阶段的核心处理节点。它的职责是基于用户输入、上游节点结果、知识库内容或工具结果生成回答。

节点说明文案：

> AI 基于检索到的知识库内容结合用户问题，生成清晰、有帮助的回答。

配置项第一阶段：

- 模型 API：选择已配置的模型供应商。
- 节点描述：展示节点用途，后续支持编辑。

配置项第二阶段：

- System Prompt。
- User Prompt。
- 上下文变量选择。
- 温度、最大输出长度、结构化输出。
- 失败重试策略。
- 输出变量名，例如 `text`、`reasoning`、`json`。

### 5.4 直接回复节点

直接回复节点负责定义最终返回给用户的内容。当前可以先展示运行调试后的 `finalOutput`，后续应升级为可配置输出模板。

配置项第二阶段：

- 回复内容模板。
- 变量插入器。
- Markdown 开关。
- 文件/图片输出。
- 多输出变量映射。

### 5.5 注释框

注释框是画布辅助信息，不进入运行 DAG，不参与保存时的边校验。

配置项：

- 标题。
- 内容。
- 颜色或标签，后续可选。

注释框用途：

- 标注流程分支目的。
- 记录测试假设。
- 写给团队协作者的说明。

## 6. 节点视觉与交互

节点卡片建议沿用当前项目的专业工具风格：白色底、轻阴影、浅边框、清晰标题、类型说明，不做过度装饰。

节点状态：

- 默认态：白底、浅边框、标题清晰。
- hover 态：轻微上浮或阴影增强，显示左右连接点。
- selected 态：使用主色描边，右侧配置面板打开。
- dragging 态：降低阴影复杂度，保持流畅。
- running 态：边框或节点顶部细进度线动效。
- failed 态：局部错误提示，不在画布节点上长期显示 success 标签。

连接点规则：

- 节点左侧为输入点，右侧为输出点。
- 连接点默认隐藏，鼠标移动到对应节点卡片时显示。
- hover 时连接点应有清晰可点区域，视觉点可以小，但命中面积要足够。
- 用户输入节点可以只有右侧输出点。
- 直接回复节点可以只有左侧输入点。
- 注释框没有连接点。

连线规则：

- 默认使用平滑曲线，从左到右连接。
- 连线应带箭头，方向表达数据流转。
- hover 连线时高亮，可用于后续删除或查看数据映射。
- 拖拽连线过程中显示临时线，落点无效时应取消。
- 删除节点时必须同步删除所有 source 或 target 指向该节点的边，保存时再次过滤悬空边。

## 7. 右侧配置面板

配置面板默认不打开。只有用户点击节点后才打开，且可以手动关闭。

面板结构：

- 顶部：节点名称、节点类型、关闭按钮。
- 主配置区：根据节点类型渲染表单。
- 运行摘要：显示最近一次运行中该节点的输入、输出、耗时、错误。
- 高级配置：后续可折叠展示重试、超时、变量输出等。

交互细节：

- 点击画布空白区域不强制关闭配置面板，避免误触导致表单丢焦。
- 关闭面板只隐藏详情，不取消节点选中也可以接受；如果视觉上保留选中，需要确保用户能理解当前节点仍被选中。
- 切换节点时，面板内容立即切换。
- 删除当前选中节点时，面板关闭。
- 用户输入默认节点不可删除，因此面板中不展示删除入口。

## 8. 数据流与变量设计

工作流本质上是有向图。节点负责声明输入、处理逻辑和输出变量；边负责表达数据流转顺序。

第一阶段数据规则：

- `nodes` 保存节点数组，包括 `id`、`type`、`name`、`description`、`position`、`config`。
- `edges` 保存边数组，包括 `id`、`source`、`target`、`sourceHandle`、`targetHandle`。
- `viewport` 保存画布缩放和平移。
- 节点删除后，前端与后端都应过滤悬空边。
- 保存时不依赖前端本地状态推断默认边，必须把用户真实连线保存下来。

变量规则第二阶段：

- 每个节点输出一个命名空间，例如 `node_llm.text`。
- 下游节点只能引用上游可达节点变量。
- 并行分支合流后，下游节点可以引用所有已合流分支输出。
- 变量选择器优先用可视化选择，文本里允许 `{{变量}}` 语法。
- 保存时同时保存变量引用的结构化表示，避免只存字符串导致后续难以校验。

## 9. 保存与后端持久化

保存按钮必须真实调用后端接口，保存用户配置的节点、边、位置和节点配置。建议后端以工作流快照为最小保存单位。

建议数据表：

### 9.1 `workflows`

| 字段 | 说明 |
| --- | --- |
| `id` | 工作流 ID |
| `agent_id` | 所属智能体 |
| `name` | 工作流名称 |
| `status` | draft / ready / blocked |
| `viewport` | 画布视口 JSON |
| `version` | 当前版本号 |
| `created_at` / `updated_at` | 时间 |

### 9.2 `workflow_nodes`

| 字段 | 说明 |
| --- | --- |
| `id` | 节点 ID |
| `workflow_id` | 所属工作流 |
| `type` | trigger / llm / expose / comment 等 |
| `name` | 节点名称 |
| `description` | 节点说明 |
| `position` | `{ x, y }` |
| `config` | 节点配置 JSON |
| `created_at` / `updated_at` | 时间 |

### 9.3 `workflow_edges`

| 字段 | 说明 |
| --- | --- |
| `id` | 边 ID |
| `workflow_id` | 所属工作流 |
| `source_node_id` | 起点节点 |
| `target_node_id` | 终点节点 |
| `source_handle` | 起点连接点 |
| `target_handle` | 终点连接点 |

### 9.4 `workflow_versions`

| 字段 | 说明 |
| --- | --- |
| `id` | 版本 ID |
| `workflow_id` | 工作流 ID |
| `version` | 版本号 |
| `snapshot` | nodes / edges / viewport / config 快照 |
| `created_by` | 操作人 |
| `created_at` | 创建时间 |

后端保存校验：

- 工作流必须至少有一个 `trigger` 节点。
- `trigger` 默认节点不可删除。
- 所有边的 source 和 target 必须存在。
- 不允许自环边。
- 第一阶段可以允许环路不存在，后续支持 loop 节点时再引入明确循环节点。
- `comment` 节点不能参与边连接。
- `llm` 节点必须配置可用模型，发布前强校验，草稿保存可弱校验。

## 10. 运行调试

运行调试用于验证当前工作流配置是否能产生预期输出。第一阶段可以只做前端按钮调用模拟接口，第二阶段应接真实执行引擎。

调试流程：

1. 用户配置用户输入节点的文本和文件输入。
2. 用户点击运行调试。
3. 前端把当前工作流快照、输入参数、选中模型、知识库等发送给后端。
4. 后端按 DAG 拓扑顺序执行节点。
5. 前端展示整体运行状态，并在节点配置面板展示选中节点的输入、输出、耗时和错误。

运行结果建议结构：

- `runId`：本次运行 ID。
- `status`：success / warning / failed / blocked。
- `steps`：每个节点的执行结果。
- `finalOutput`：最终回复。
- `cost`：费用。
- `latencyMs`：总耗时。

画布运行态：

- 正在运行的节点显示细线进度或轻微边框动效。
- 执行完成不显示永久 success 标签，避免画布视觉噪音。
- 失败节点用红色边框和右侧详情错误文案表达。

## 11. 后续节点扩展路线

建议按业务价值逐步扩展：

第一阶段：

- 用户输入。
- LLM。
- 直接回复。
- 注释框。

第二阶段：

- 知识检索节点：选择知识库、检索 query、topK、重排、元数据过滤。
- 工具节点：选择 MCP/API 工具，配置入参映射和权限。
- 条件节点：按变量、字符串、数值、布尔判断分支。

第三阶段：

- 人工确认节点：暂停执行，等待人工审批或补充输入。
- 变量赋值节点：把上游结果写入工作流变量或会话变量。
- 代码节点：执行轻量脚本，处理结构化数据。

第四阶段：

- 迭代节点：对数组逐项处理。
- 循环节点：支持带退出条件的循环。
- 子工作流节点：复用其他工作流作为工具。

## 12. 验收标准

基础画布：

- 工作流页主画布占满主卡片区域。
- 左侧五个工具可见且功能明确。
- 点击添加 LLM 后，需要用户手动点击画布决定节点位置。
- 点击添加注释框后，需要用户手动点击画布决定注释位置。
- 指针模式支持选中、拖拽、连线。
- 手模式支持画布拖动，不误拖节点。
- 自动整理按从左到右重新排列节点。

节点与配置：

- 每个智能体默认有用户输入节点。
- 用户输入节点不可删除。
- 用户输入节点有文字输入和文件上传输入两个配置项。
- 点击节点才打开右侧配置面板。
- 配置面板可关闭。
- LLM 节点可新增、配置、删除。
- 删除节点后，对应边也删除，保存不失败。

连线与保存：

- 节点连接点默认隐藏，hover 对应节点时显示。
- 连线从右侧输出点到左侧输入点。
- 保存后刷新页面，节点位置、节点配置、连线保持一致。
- 保存 payload 不包含指向已删除节点的边。

运行调试：

- 用户能输入测试文本并运行。
- 运行结果展示在右侧详情或结果区。
- 失败时给出可理解错误，不阻断继续编辑。

## 13. 设计原则

- 画布是主角，配置面板是上下文辅助，不应默认占据注意力。
- 连线代表真实数据流，系统不暗中替用户连接。
- 默认节点少而可用，后续节点逐步补齐。
- 节点状态只在需要时显示，避免每个节点长期挂 success 标签。
- 前端交互要即时，后端保存要严谨，最终以后端校验为准。
- 所有节点配置都要能保存、复现、运行、追踪。
